### 第1,2回
　ソフトウェア工学は、「品質」「コスト」「納期」の最適なバランスを実現するための手法である。単なるプログラミングとは異なり、開発・保守の「スケール」や、ライブラリやOSの更新といった時間軸における持続可能性の検討が重要である。
　実践の本質は、問題の理解、計画、実装、テストの4工程にあり、開発工程が本番環境へ進むほど欠陥のコストが大きく増大するため、早期の対応が重要である。また、システムはユーザーへの価値提供のためにあるという「Hookerの7原則」や、仕様に関わらず観察可能なあらゆる挙動にユーザーが依存するという「Hyrumの法則」に基づいて、変化し続ける環境へ適応する安全なコードベースの維持が求められる。

### 第3回
　ソフトウェアライフサイクルは、企画や要件定義を行う「誕生」、開発や保守運用を担う「開発・運用」、そして「廃止」のフェーズに分かれる。要件定義は、目的や仕様を明文化して関係者間でスコープを共有するバイブルとしての意義を持つ。続く設計を基にシステムを構築する制作フェーズでは、内製のほか外部委託（調達）も選択され、海外へ委託するオフショア開発では文化や品質意識の違いへの注意が必要である。　
　テストではバグの早期発見が理想とされるが工数に限界があり、運用・保守では初期故障や安定稼働といったバスタブ曲線に応じた対応が求められる。また、複数人での開発においては、ドキュメンテーション（非同期な文書によるコミュニケーション）が重要である。

### 第4回
　プロジェクトは有期性と独自性を持ち、目標から逆算して計画する「バックキャスティング」が適している。ソフトウェア分析は、コードの物量やファンクションポイント法による規模見積もり、品質管理（バグ発生率など）を通じてQCDの最適化する手法である。
　開発プロセスには、進捗管理が容易だが後工程に負担が集中しやすいウォーターフォール型（計画駆動型）のほか、プロトタイプを繰り返すスパイラルモデル、機能分割して積み上げる反復型、変化への対応を最優先するアジャイルプロセスがある。アジャイルの手法としてスクラムやカンバン、XP、DevOpsなどがある。手法はそれぞれ一長一短であり、開発規模、変化の度合い、開発者のスキルや組織文化、リスクの特性に応じて場面ごとに使い分けることが求められる。

### 第5,6回
　WBSとは、プロジェクト目標の達成と必要な成果物の生成に向け、作業を階層的に要素分解した構成図である 。作成によりスコープややるべき作業が明確になり、全体管理を容易にするメリットがある 。作り方の手順として、バックキャスティングの視点からトップダウンでスコープを明確にし、大きな作業をグルーピングした上で、その相互関連を考慮しながらMECE（もれなくダブりなく）に各タスクを洗い出す手法が挙げられる。
　課題では上位2人が勝ち上がる麻雀大会を開催する想定で、誰が行っても同様にできるように、準備から開催日までのWBSと大会当日の行程をまとめたWBSをそれぞれ作成した。

### 第7回
　コードは書くよりも読まれる回数の方が多いため、他者や未来の自分が読みやすい「可読性」を重視したルールが必要となる。Pythonの哲学である「The Zen of Python」を踏まえ、コードレイアウト（1行79文字以内、スペース4つのインデント、適切な空白や改行、インポートの順序、最新に保つコメント、命名規則）を守ることで可読性の高いコードを書くことが出来る。また、最も重要なのは、プロジェクト内での一貫性を保つことである。
　「flake8」は規約に基づくコードチェックツール（Linter）である。課題では仮想環境を構築し、サンプルコードを実行して検出された各種のエラーを解決した。

### 第8,9,10回
　Gitのブランチ操作を学習するため、「Learning Git Branching」を活用した。ここではコミットやクローン、HEADの分離や「cherry-pick」などを学習した。
　また、様々なgitコマンドには様々な種類がある。それらはリポジトリの初期化や履歴確認を行う設定・確認系、ステージングと記録を担う add や commit などのコミット系、状態の復元やリセットを行う修正系、リモートリポジトリとの同期やアップロードを行う pull や push などのリモート系、そして分岐を制御するブランチ系に分類される。

### 第11回
　GitHubは、Gitを用いたローカルリポジトリと連携するリモートリポジトリのホスティングサービスであり、オープンソースソフトウェア開発の主要なポータルサイトとして機能している。
　共同開発を支える主要なリポジトリ操作として、他ユーザーのリポジトリを自身の元のリモート環境へ複製する「fork」や、リモートリポジトリを自身のローカル環境へコピーする「clone」、リポジトリの構成を流用して新しいリポジトリを作成できる「template」機能が提供されている。
　また、プロジェクトの管理機能として「issues」と「projects」がある。issuesはプログラムのバグ、課題、機能追加の要望、質問などをチケットとして管理する役割を持ち、projectsはこれらの課題をテーブル、kanbanを用いてプロセスごとに整理・追跡するための進捗管理ツールである。
　実際の開発現場では「GitHubフロー」と呼ばれるブランチ運用モデルが用いられている。開発者は本番用の main ブランチから機能開発や修正用に dev などの個別ブランチを分岐させ、実装を重ねた後に「Pull Request」を通じて変更依頼を出す。これにより他ユーザーとのコードレビューや変更内容の確認を経た上で、安全に main ブランチへ変更を統合（merge）することが可能となる。
　課題では実際にGitHubフローに則り、ソフトウェア工学の内容をまとめた内容のmdファイルを作成した。

### 第12,13回
　CI/CD（Continuous Integration/Continuous Delivery）は、現代のソフトウェア開発において開発速度の向上とコードの品質保証を両立するための重要な手法である。
　CIとはコード変更を頻繁にリポジトリに統合して自動テストやビルドを行うプロセスであり、CDはテスト済みのコードを自動で本番環境へデプロイするプロセスを指す。この自動化にはGitHub Actionsが用いられ、指定のディレクトリにYAML形式の設定ファイルを配置することで、プルリクエスト等をトリガーにジョブやシェルコマンドを自動実行させることができる。さらに、GitHub PagesとJekyllを組み合わせることで、開発者が作成したマークダウンファイルを自動でHTMLへとビルドし、そのままWeb上に公開・更新する環境を構築できる。これらにより、ソースのビルド、テスト、デプロイ、検証にいたる一連のリリースパイプラインをシームレスに自動化することが可能となる。
